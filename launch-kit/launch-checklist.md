# One-page launch checklist

## Before launch

- [ ] Confirm `main` is green: `python3 scripts/run-evals.py` reports `108/108`.
- [ ] Confirm GitHub Actions badge is green on the public repository.
- [ ] Apply `github-polish.md`: description, seven topics, social preview, and star line.
- [ ] Confirm README install commands work from a clean Claude Code configuration.
- [ ] Add one tracked GitHub issue labeled `feedback wanted` for bypass reports.
- [ ] Prepare a dated metrics sheet: stars, unique cloners, referring sites, issues, contributors, and marketplace status. Record zeros; never backfill guesses.
- [ ] Check every screenshot at desktop and mobile crop sizes.
- [ ] Verify every public link in a logged-out browser.

## Post order

| Order | Channel | Timing | Purpose |
| --- | --- | --- | --- |
| 1 | GitHub polish | T−24h | Make the destination credible before sending traffic. |
| 2 | Show HN | T0, weekday morning US ET | Invite technical scrutiny and bypass reports first. |
| 3 | Product Hunt | Same day, after HN is stable | Broader discovery with proof-rich gallery. |
| 4 | LinkedIn short post | T+2–4h | Connect the open-source proof to consulting credibility. |
| 5 | X technical thread | T+4–6h | Explain the audit/verdict mechanics and drive developer feedback. |
| 6 | Dev.to / Hashnode | T+1 day | Create a durable, searchable technical narrative. |
| 7 | LinkedIn article | T+2–3 days | Expand the consulting story after discussion reveals objections. |
| 8 | X business thread | T+3–5 days | Translate the system into agency/operations value without invented savings. |
| 9 | Awesome-list PRs | T+5–10 days | Submit separately, following each repository's contribution rules. |

## Launch-day operating loop

- [ ] Reply to technical criticism with a fixture or issue, not a defensive claim.
- [ ] Convert each reproducible bypass into a failing test before patching it.
- [ ] Answer “does it prevent hallucinations?” with the qualified ship-gate explanation in `product-hunt.md`.
- [ ] Keep the CTA soft: try it, star it if useful, report a bypass, or ask Corriston Consulting about a proof-first production system.
- [ ] Do not cross-post identical text; use the channel-specific drafts.

## Measurement checkpoints

- [ ] T+24h: record sourced metrics and top three objections.
- [ ] T+7d: publish only real, dated adoption numbers; summarize accepted issues and merged contributions.
- [ ] T+30d: write a follow-up only if there is a defensible signal—usage, a case study, contributor activity, or a meaningful validator improvement.
- [ ] Replace no draft claim with an estimate. Use “not yet measured” when the number does not exist.
